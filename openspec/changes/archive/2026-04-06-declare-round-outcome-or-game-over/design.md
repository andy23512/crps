## Context

The current gameplay flow has two core player actions per round: select a move and view result. The latest implementation already reveals the computer move before player selection. This change adds a stricter challenge contract: the application randomly generates a required target outcome (win, lose, or draw) each round, and failing to achieve that target immediately ends the game.

## Goals / Non-Goals

**Goals:**

- Add a mandatory per-round target generation step before move selection
- Evaluate each completed round against both normal RPS resolution and the generated target outcome
- Enter a game-over state when declaration and actual result do not match
- Prevent further round play while game-over is active until reset/new game
- Keep existing move validity and RPS winner logic unchanged

**Non-Goals:**

- Changing rock/paper/scissors resolution rules
- Changing randomness model of computer move generation
- Adding backend persistence, user accounts, or multiplayer behavior
- Adding alternate failure penalties beyond game over

## Decisions

### Decision 1: Store generated target outcome in round state

Each active round tracks one target outcome generated by the application from {win, lose, draw}.

Rationale:

- Keeps rule evaluation explicit and deterministic
- Makes validation and UI state transitions straightforward

Alternative considered:

- Generate target only for display and regenerate at resolution time. Rejected because it breaks fairness and predictability.

### Decision 2: Game-over is terminal until reset

When the actual round result does not match the generated target outcome, set `isGameOver=true`, display game-over status, and disable move actions.

Rationale:

- Directly satisfies the requested rule "or the game will over"
- Prevents accidental continuation after failure

Alternative considered:

- Allow continuation with penalty. Rejected because it conflicts with required immediate game-over behavior.

### Decision 3: Preserve score consistency at failure boundary

Score updates only for the completed round result, then game-over lock applies to future rounds until reset.

Rationale:

- Avoids ambiguous partial-round updates
- Preserves existing scoring semantics while adding terminal-state gating

## Risks / Trade-offs

- [Extra state can make flow less obvious] -> Mitigation: keep generated target prominently visible before move selection.
- [Players may misread required target] -> Mitigation: show clear, high-contrast target label and current-round context.
- [State bugs could allow play after game over] -> Mitigation: gate move/declaration handlers on `isGameOver` and cover with tests.

## Migration Plan

1. Add generated target outcome and game-over fields to game state model/service.
2. Add UI display for required target outcome generated by the app.
3. Update round flow to validate result against generated target and toggle game-over when mismatch occurs.
4. Disable play controls after game-over and show terminal message.
5. Reset flow clears scores, target, result, and game-over flag for fresh start.
6. Verify matching/mismatching generated-target scenarios and post-game-over lockout.

Rollback strategy:

- Remove declaration/game-over gating and restore current revealed-move round flow.

## Open Questions

- Should a new target always be generated immediately after each successful round?
- Should the final failed round still be shown in history/result panel after game over (recommended yes for clarity)?
